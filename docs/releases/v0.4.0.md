# Faultline v0.4.0 (Planned)

Faultline v0.4.0 is planned as the release that extends the shipped
deterministic CLI into fleet-safe deterministic operations while preserving the
current local-first, audit-friendly release boundary.

This is a roadmap document, not a shipped release note. It describes the
intended order of work and the release guardrails that should hold while the
release is being built.

## Release Theme

**Fleet-safe deterministic operations**

The release should make four things clear:

- the deterministic forensic engine remains the substrate for every later
  feature
- managed inheritance is the main enterprise headline
- reliability metrics and quarantine remain additive policy outputs
- the default user story stays anchored to the existing CLI core

## Ordered Delivery

1. ✓ Harden the deterministic forensic engine around `analyze`, `workflow`,
   `fix`, `trace`, differential diagnosis, and stable JSON
2. ✓ Land managed inheritance through the existing `packs` boundary with local
   sync, pinned references, and pack provenance
3. ✓ Add a hidden maintainer-only authoring assistant that composes with the
   sanitizer, fixtures pipeline, and review gates
4. ✓ Expose PHI, TSS, and FPC as additive machine-readable metrics derived from
   explicit artifact sets or supplied history
5. ✓ Emit advisory quarantine policy based on those metrics without taking over
   retries or CI orchestration

## Boundary Status

The default shipping narrative for v0.4 should remain:

- `analyze`
- `workflow`
- `list`
- `explain`
- `fix`

Everything else should remain companion, hidden, or explicitly flagged until it
meets the same deterministic validation standard as the existing shipped path.

That means:

- no runtime network fetch for `analyze`, `workflow`, or `trace`
- no AI or LLM logic in the product's authoritative matching or policy path
- no dashboards or hosted storage added to the core product shape
- no quarantine-driven retries or suite routing inside Faultline itself

## Planned Interface Additions

v0.4 is expected to grow the public artifact contracts additively:

- `packs` metadata for synced references, pinned versions, and inherited pack
  lineage — **landed**: `InstalledPack` carries `Version`, `PinnedRef`, and
  `SourceURL`; `faultline-pack.yaml` manifests record provenance at install
  time; `packs list` shows VERSION and PINNED REF columns; analysis JSON
  includes `pack_provenance`
- `fixtures scaffold` (hidden, maintainer-only): generates a candidate playbook
  YAML from a sanitized log with deterministic pattern extraction; composes with
  `fixtures sanitize`, `fixtures review`, and `fixtures promote`
- analysis JSON sections for `metrics` and `policy` — **landed**: `Analysis`
  carries additive `metrics` (TSS, FPC, PHI, drift components) and `policy`
  (`recommendation`, `reason`, `basis`) fields; both are omitted when
  insufficient history is available
- workflow JSON hints derived from the same deterministic result object —
  **landed**: `metrics_hints` and `policy_hints` arrays appear in the workflow
  JSON artifact when metrics and policy are computed; both are omitted when
  absent

Absent data should stay absent from output. The CLI should not infer or invent
values when the required artifact history is not available.

## Validation Direction

Before any v0.4 feature graduates into the shipped path, it should be defended
with repository-native proof:

- repeated-run determinism checks for output and pack provenance
- offline resolution checks for synced packs after an explicit sync step
- fixture-backed review for authoring changes
- snapshot-tested metric calculations and threshold handling
- policy tests proving quarantine stays advisory
