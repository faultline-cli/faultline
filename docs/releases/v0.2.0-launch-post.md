# Launch Draft: Deterministic CI Failure Diagnosis

We built a deterministic CI failure engine that can replay and compare failing builds locally.

No AI.
No guessing.
Same input, same output.

## The problem

CI failures are expensive mostly because diagnosis is noisy and inconsistent.

Teams scroll long logs, argue over likely causes, and rerun pipelines to verify guesses. Most tooling optimizes for plausible explanations. Very little tooling optimizes for deterministic, reproducible diagnosis.

## What Faultline does

Faultline analyzes failing CI logs with checked-in detectors and fixture-validated playbooks.

It returns:

- a ranked diagnosis
- evidence lines pulled directly from the log
- deterministic follow-through workflow steps

The core behavior is local-first and deterministic by default.

## Why this is different

Faultline is designed as infrastructure, not a chat assistant:

- deterministic detection stays authoritative
- optional Bayes-inspired reranking stays additive and explainable
- stable JSON contracts support automation and agent workflows
- fixture-backed regression gates protect release quality

If a diagnosis changes, you can trace exactly why.

## What is already built

Faultline ships with a default stable path and additional complete surfaces:

- Stable default: `analyze`, `workflow`, `list`, `explain`, `fix`
- Complete companion surfaces: `trace`, `replay`, `compare`, `inspect`, `guard`
- Experimental provider-backed delta: `--delta-provider github-actions` behind `FAULTLINE_EXPERIMENTAL_GITHUB_DELTA=1`

This allows controlled visibility without weakening release discipline.

## Real workflow example

In a failing GitHub Actions run:

```bash
faultline analyze build.log --format markdown --ci-annotations > faultline-summary.md
faultline analyze build.log --json --bayes > faultline-analysis.json
faultline workflow build.log --json --mode agent > faultline-workflow.json
```

Result: deterministic diagnosis for humans and structured artifacts for automation.
