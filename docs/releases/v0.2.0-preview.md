# Faultline v0.2.0 Preview

Faultline v0.2.0 is about proving that deterministic CI diagnosis can be part of a real automation loop, not just a one-off log reader.

## Positioning

Faultline is a deterministic prevention, diagnosis, and remediation handoff layer for CI/CD:

- deterministic detection stays authoritative
- evidence comes directly from matched log lines
- fixes come from checked-in playbooks
- workflow artifacts stay stable enough for scripts, agents, and CI summaries

That makes the product audit-friendly, procurement-friendly, and safe to automate against for teams that care about repeatability.

## What This Slice Proves

- `workflow` is now promoted alongside `analyze` in the main product narrative.
- The repository ships checked-in workflow example artifacts for human and agent consumption.
- The fixture corpus is exposed as a public trust artifact instead of staying implicit.
- The release story is framed around deterministic prevention, diagnosis, and remediation handoff.

## Proof Points In This Repository

- 74 bundled playbooks
- 84 accepted real fixtures
- 84/84 top-1 on the accepted real-fixture corpus
- 84/84 top-3 on the accepted real-fixture corpus
- 0 unmatched fixtures in the accepted real-fixture corpus
- 0 false positives in the accepted real-fixture corpus
- 9 weak matches still tracked in the current baseline

See `docs/fixture-corpus.md` for the current snapshot and regeneration commands.

## Operator-Facing Additions

- `workflow` now ships as a first-class artifact alongside `analyze`, with checked-in local and agent snapshots in `examples/`.
- `fix`, `inspect`, `guard`, and `packs` extend the deterministic loop from diagnosis into remediation, local prevention, and catalog composition.
- `analyze` now supports `--ci-annotations` plus opt-in `--delta-provider github-actions` for provider-backed failure-delta context without changing the deterministic default path.
- The repository now publishes both task-specific prompts and five repo-local agent skills for fixture curation, coverage review, playbook refinement, new playbook authoring, and baseline regression handling.

## Release Gates

The release candidate path for this slice should stay explicit:

- `make release-check VERSION=v0.2.0`
- `WITH_DOCKER=1 IMAGE=faultline-v0.2.0 make release-check VERSION=v0.2.0` when the image contract changed

This keeps tests, playbook overlap review, fixture regression, archive packaging, and smoke validation coupled to the release cut instead of relying on manual spot checks.

## Workflow As A First-Class Surface

Faultline does more than label the failure. It also emits a deterministic next-step artifact:

- local text workflow output for direct engineering follow-through
- agent-oriented JSON workflow output for scripts and agent prompts

The checked-in snapshots for the missing-executable example live in:

- `examples/missing-executable.workflow.local.txt`
- `examples/missing-executable.workflow.agent.json`

## What Stays True In v0.2.0

- same input and same checked-in rule set produce the same output
- no ML or LLM logic enters the core product path
- playbook matching and fix content remain explicit and reviewable
- Bayesian-inspired scoring is assistive only: it reranks detected candidates and ranks likely drift causes without replacing deterministic matching
- automation contracts are derived from stable CLI output, not provider-specific branches in the core runtime
