# Faultline v0.3 CLI UX Specification

> Product behavior proposal only. [`SYSTEM.md`](../../SYSTEM.md) remains authoritative for implementation boundaries and invariants.

## 1. Executive Summary

Faultline v0.3 should become more inspectable and more polished without changing its trust model.

The right move is not a full-screen TUI. The right move is a better CLI:

- keep `analyze`, `workflow`, `list`, `explain`, and `fix` as the default story
- add explicit drill-down surfaces for trace, comparison, and replay
- make terminal output calmer, denser, and easier to scan under pressure
- make markdown output first-class for CI summaries, GitHub comments, and issue handoff
- make playbooks feel like debuggable programs through trace, lint, preview, and targeted tests

The product posture should stay narrow and high-trust:

- deterministic matching remains authoritative
- optional interactivity is additive and TTY-only
- every interactive affordance must degrade to a normal command or a flag
- JSON stays stable and scriptable
- markdown stays deterministic and paste-friendly

The practical v0.3 theme is:

**summary first, evidence second, trace third, raw details last**

## 2. Top Enhancements

Ranked highest impact first.

1. Introduce a first-class trace surface: `faultline trace <logfile>` and `faultline analyze --trace`.
2. Redesign default terminal output around a compact summary card plus structured evidence and next actions.
3. Add explicit candidate drill-down via `--select <rank>` and focused views via `--view summary|evidence|trace|fix|raw`.
4. Ship a premium markdown mode intended for GitHub comments, CI summaries, and copied issue reports.
5. Add rejected-match visibility with `--show-rejected` and targeted trace filtering with `--trace-playbook <id>`.
6. Separate remediation into `primary fix`, `investigate next`, and `verify` so operators can act with calibrated confidence.
7. Add playbook authoring commands: `faultline playbook lint|test|trace|preview|doctor`.
8. Add deterministic replay from saved analysis artifacts: `faultline replay <analysis.json>`.
9. Add deterministic run comparison: `faultline compare <left> <right>`.
10. Improve `explain` so it shows related playbooks, disproof checks, and trace-ready rule labels.
11. Add pager-aware long output with `--page auto|always|never` and conservative TTY detection.
12. Add a lightweight interactive inspect mode behind an experimental flag for supported terminals only.
13. Extend playbook schema so rule items can carry labels and notes, making trace output human-readable.
14. Add authoring diagnostics for overlap, ambiguity, weak evidence, and poor remediation quality.
15. Add stable provenance metadata to markdown and replay artifacts.

## 3. Proposed CLI Spec

### 3.1 Command Surface

Keep the default onboarding path unchanged:

```text
faultline analyze [file]
faultline workflow [file]
faultline fix [file]
faultline list
faultline explain <id>
```

Add advanced but deterministic companion surfaces:

```text
faultline trace [file]
faultline replay <analysis.json>
faultline compare <left> <right>
faultline playbook lint [path...]
faultline playbook test <playbook-or-dir> [fixture...]
faultline playbook trace <id> [file]
faultline playbook preview <id>
faultline playbook doctor [path...]
```

Keep existing companions:

```text
faultline inspect [path]
faultline guard [path]
faultline packs ...
```

Do not add a standalone `faultline render markdown` command in v0.3.

- `faultline analyze --format markdown` should remain the primary path.
- `faultline replay <analysis.json> --format markdown` should cover artifact rerendering.
- This keeps the command surface smaller and avoids a redundant rendering-only top-level verb.

### 3.2 Command Roles

#### `faultline analyze [file]`

Purpose:
- primary diagnosis command

User value:
- fastest path from failing log to likely diagnosis, evidence, and next action

CI safety:
- safe by default

Determinism:
- unchanged; optional flags only change rendering or reveal already-computed detail

Recommended new flags:

- `--view summary|evidence|trace|fix|raw`
  Purpose: focus one slice of the result without dumping everything.
  CI-safe: yes.
  Determinism: pure presentation.
- `--select <rank>`
  Purpose: inspect a specific ranked result.
  CI-safe: yes.
  Determinism: selects an already-ranked candidate.
- `--trace`
  Purpose: append trace detail for the winning playbook.
  CI-safe: yes.
  Determinism: exposes evaluation detail only.
- `--trace-playbook <id>`
  Purpose: show trace only for one named playbook.
  CI-safe: yes.
  Determinism: scoped inspection only.
- `--show-rejected`
  Purpose: show near matches and explicitly rejected playbooks.
  CI-safe: yes.
  Determinism: exposes already-known rejection reasons.
- `--show-evidence`
  Purpose: expand supporting evidence lines even in summary mode.
  CI-safe: yes.
  Determinism: pure presentation.
- `--show-scoring`
  Purpose: include score and confidence breakdowns.
  CI-safe: yes.
  Determinism: pure presentation.
- `--page auto|always|never`
  Purpose: send long output through a pager when appropriate.
  CI-safe: yes; default `auto` disables paging outside a TTY.
  Determinism: presentation only.
- `--theme compact|rich|plain`
  Purpose: choose visual density.
  CI-safe: yes; `plain` remains safest fallback.
  Determinism: presentation only.
- `--interactive`
  Purpose: lightweight drill-down in a capable TTY.
  CI-safe: no practical value in CI; should auto-disable or error on non-TTY.
  Determinism: navigation only; no extra inference.

Recommended defaults:

- default view: `summary`
- `--show-evidence`: off by default
- `--show-scoring`: off by default unless `--bayes`
- `--page`: `auto`
- `--interactive`: off

#### `faultline trace [file]`

Purpose:
- dedicated execution transparency view for evaluation details

User value:
- answers "why this matched", "why this lost", and "what almost matched"

CI safety:
- yes; especially useful as an artifact command

Determinism:
- trace is a deterministic projection of the same matcher and scorer state

Key flags:

- `--playbook <id>`
- `--top <n>`
- `--show-rejected`
- `--show-evidence`
- `--show-scoring`
- `--format terminal|markdown|json`

Design note:
- `trace` should be the full view
- `analyze --trace` should be the shortcut

#### `faultline workflow [file]`

Purpose:
- turn the top diagnosis into next-step execution guidance

User value:
- handoff from diagnosis to action without inventing workflow glue

CI safety:
- yes

Determinism:
- unchanged

Recommended refinements:

- split output into `focus`, `remediation`, and `verification`
- add `--checklist` to render numbered operator checklists
- add `--commands-only` for copy-paste-focused output
- add `--after-fix` to include the recommended re-run command or verification set

These are safe for CI and deterministic because they only reshape known workflow data.

#### `faultline fix [file]`

Purpose:
- fastest remediation-only path

User value:
- when the diagnosis is already trusted and the operator only needs the fix

CI safety:
- yes

Determinism:
- unchanged

Recommended refinements:

- render `primary fix`, `investigate next`, and `verify`
- add `--commands-only`
- add `--with-preconditions`
- add `--with-risks`

#### `faultline explain <id>`

Purpose:
- inspect a playbook as a catalog entry

User value:
- understand intent, coverage boundary, and remediation content

CI safety:
- yes

Determinism:
- unchanged

Recommended refinements:

- show concise rule labels instead of raw pattern lists when available
- include `related playbooks`
- include `common confusions`
- include `disproof checks`
- add `--trace-sample <file>` to render this playbook against a log without changing commands

#### `faultline compare <left> <right>`

Purpose:
- compare two deterministic analyses or two logs

User value:
- answer why a different playbook won this time, what evidence changed, and whether the environment drifted

CI safety:
- yes

Determinism:
- yes; compares explicit inputs or saved artifacts

Accepted inputs:

- two analysis JSON artifacts
- two markdown/terminal replay artifacts plus embedded metadata
- two raw logs
- one raw log and one saved analysis artifact

Key views:

- diagnosis delta
- evidence delta
- rejected-candidate delta
- repo-context delta
- environment delta

Recommended flags:

- `--view summary|evidence|diagnosis|env|full`
- `--new-evidence-only`
- `--why-changed`
- `--format terminal|markdown|json`

Status:
- advanced companion surface
- ship behind `--experimental-compare` in alpha, default-visible by beta

#### `faultline replay <analysis.json>`

Purpose:
- re-render a saved analysis artifact without re-running matching

User value:
- stable report reproduction for CI artifacts, reviews, and docs

CI safety:
- yes

Determinism:
- yes; output is derived from a frozen artifact

Recommended flags:

- `--format terminal|markdown|json`
- `--view summary|evidence|trace|fix|raw`
- `--select <rank>`

#### `faultline playbook ...`

Keep the group out of the first-run narrative. This is the authoring and maintenance surface.

`faultline playbook lint [path...]`
- Purpose: schema, metadata, wording, and remediation quality lint.
- CI-safe: yes.
- Determinism: yes.

`faultline playbook test <playbook-or-dir> [fixture...]`
- Purpose: fixture-backed playbook checks with expected winner assertions.
- CI-safe: yes.
- Determinism: yes.

`faultline playbook trace <id> [file]`
- Purpose: deep trace for one playbook against one input.
- CI-safe: yes.
- Determinism: yes.

`faultline playbook preview <id>`
- Purpose: preview `terminal`, `markdown`, and `fix` renderings from playbook data.
- CI-safe: yes.
- Determinism: yes.

`faultline playbook doctor [path...]`
- Purpose: overlap, ambiguity, weak evidence, low-quality remediation, and missing verification diagnostics.
- CI-safe: yes.
- Determinism: yes.

### 3.3 Aliases And Discoverability

Use aliases sparingly. Faultline should feel serious, not shell-clever.

Recommended aliases:

- `faultline ls` → `faultline list`
- `faultline pb` → `faultline playbook`

Do not add cute one-letter aliases.

Improve discoverability through help structure instead:

- root help grouped into `Diagnose`, `Inspect`, `Author`, and `Catalog`
- every command shows one local example and one CI-friendly example
- `faultline analyze --help` shows a "drill down" section for `--view`, `--select`, and `--trace`
- `faultline playbook --help` stays visible but clearly marked as advanced

### 3.4 Defaults, Opt-In, Experimental

Default:

- improved compact terminal output
- `--view summary`
- `--page auto`
- richer markdown renderer
- `trace` command
- `replay` command
- `playbook lint`, `playbook test`, `playbook preview`

Opt-in but stable:

- `--trace`
- `--show-rejected`
- `--show-evidence`
- `--show-scoring`
- `--select`
- `--view trace`
- `compare`
- `playbook trace`
- `playbook doctor`

Experimental:

- `--interactive`
- keyboard navigation in supported terminals
- compare auto-discovery against local history store
- collapsible markdown details

## 4. Output Design Examples

### 4.1 Default Terminal Mode

Goal:
- one screen first
- diagnosis, evidence, action, and confidence without noise

```text
FAULTLINE  analyze  examples/missing-executable.log

Most likely diagnosis
---------------------
missing-executable  Required executable or runtime binary missing
Confidence: high (84%)   Category: build   Severity: high

Summary
-------
The job tried to launch a required tool or runtime binary, but that executable
was missing from the image, runner, or expected path.

Matched evidence
----------------
1. exec /__e/node20/bin/node: no such file or directory

Recommended action
------------------
Primary fix:
1. Install the missing runtime or tool in the CI image.
2. Pin the runner or action to an image that includes the expected binary.

Verify:
1. command -v <tool>
2. Re-run the failing workflow step.

More
----
Use `faultline analyze examples/missing-executable.log --trace`
Use `faultline workflow examples/missing-executable.log`
Use `faultline analyze examples/missing-executable.log --select 2 --view summary`
```

Design notes:

- no decorative box maze
- restrained rules and spacing
- confidence phrased as text plus percentage
- only one evidence section by default
- next-step hints are verbs, not marketing copy

### 4.2 Expanded Interactive Mode

This is additive, not the default interface.

Invocation:

```bash
faultline analyze ci.log --interactive
```

Behavior:

- only enabled on a real TTY
- does not take over the full screen
- uses a pager-style detail loop
- every key action maps to a normal command

Example:

```text
FAULTLINE  analyze  ci.log

1. missing-executable   84%
2. arch-mismatch        41%
3. runtime-mismatch     37%

View: summary   Candidate: #1
Keys: 1-3 select  e evidence  t trace  f fix  r rejected  q quit

Summary
-------
The job tried to launch a required tool or runtime binary, but that executable
was missing from the image, runner, or expected path.

Matched evidence
----------------
1. exec /__e/node20/bin/node: no such file or directory

[t]

Trace: missing-executable
-------------------------
PASS  match.any[missing_runtime_path]
      pattern: exec /__e/node20/bin/node: no such file or directory
      evidence: line 182
      note: GitHub Actions runtime shim points at a missing binary path

FAIL  match.none[permission_denied_variant]
      pattern: permission denied
      note: not present, so the permission-denied branch is not active

WIN   why this won
      direct path-missing evidence matched exactly
      runner-up only matched generic runtime failure wording
```

Fallback behavior:

- non-TTY: error with guidance to use `--view` and `--select`
- minimal terminals: fall back to plain text
- `NO_COLOR`: disable color, keep structure

Intended for GitHub, CI summaries, and issue trackers.

## Most Likely Diagnosis

## Why It Matched

- Exact evidence matched a missing runtime path failure.
- No contradicting permission-denied evidence was present.
- Nearby runtime playbooks matched less specific wording only.

## Evidence

```text

```

## Recommended Actions

### Primary Fix

1. Install the missing runtime or tool in the CI image.
2. Pin the runner or action to an image that includes the expected binary.

### Verification

1. Run `command -v <tool>`.
2. Re-run the failing workflow step.

## Matching Playbooks

1. `missing-executable` - 84%
2. `arch-mismatch` - 41%
3. `runtime-mismatch` - 37%

## Metadata

- Generated by Faultline `v0.3.0-beta`
- Rendering mode: `markdown/compact`
- Deterministic input fingerprint recorded above
```

### 4.4 Playbook Trace Mode

```text
TRACE  missing-executable  [log detector]
Source: examples/missing-executable.log
Outcome: matched and ranked #1

Rule evaluation
---------------
PASS  any.missing_runtime_path
      pattern: exec /__e/node20/bin/node: no such file or directory
      evidence:
        - line 182: exec /__e/node20/bin/node: no such file or directory
      relevance: strong
      note: exact path-missing signature

PASS  any.command_not_found
      pattern: no such file or directory
      evidence:
        - line 182: exec /__e/node20/bin/node: no such file or directory
      relevance: supporting

FAIL  none.permission_denied_variant
      pattern: permission denied
      evidence: none
      note: exclusion not triggered

Score
-----
Base score: 1.60
Compound bonus: +0.50
Final score: 2.10
Confidence: 84%

Why this won
------------
- exact runtime path failure matched
- no exclusion triggered
- alternatives matched only generic runtime symptoms

Rejected alternatives
---------------------
arch-mismatch
- rejected from top spot because no architecture mismatch evidence was present

runtime-mismatch
- rejected from top spot because the log showed path absence, not version conflict
```

## 5. Markdown Aesthetics Spec

### 5.1 Goals

Markdown output must be:

- clean in GitHub
- readable when pasted into issue trackers
- stable and deterministic
- readable both raw and rendered
- easy to diff in snapshots

### 5.2 Section Order

Use this exact order for analysis markdown:

1. `# Faultline Analysis`
2. `## Summary`
3. `## Most Likely Diagnosis`
4. `## Confidence And Evidence`
5. `## Matching Playbooks`
6. `## Why It Matched`
7. `## Recommended Actions`
8. `## Verification`
9. `## Alternatives And Rejections` when requested
10. `## Raw Evidence Appendix` when requested
11. `## Metadata`

Trace markdown uses:

1. `# Faultline Trace`
2. `## Outcome`
3. `## Rule Evaluation`
4. `## Score`
5. `## Why This Won`
6. `## Rejected Alternatives`
7. `## Metadata`

### 5.3 Formatting Rules

- headings only; no HTML-heavy layout by default
- bullets for metadata and rationale
- numbered lists for remediation and verification steps
- code fences for evidence excerpts and shell commands
- tables only for compact rank lists or compare output where alignment matters
- one blank line between sections
- keep line width conservative for raw markdown readability

### 5.4 Collapsible Sections

Support collapsible sections only in `--markdown-mode rich`, never in default markdown.

Allowed sections:

- raw evidence appendix
- rejected alternatives
- full trace detail

Implementation rule:

- default markdown must remain pure, stable, and renderer-agnostic
- collapsible details should emit GitHub-friendly `<details>` blocks only when explicitly requested

### 5.5 Tone

Use a restrained technical voice:

- "Most likely diagnosis"
- "Why it matched"
- "Recommended actions"
- "Verification"

Avoid:

- celebratory language
- emoji as structural markers
- excessive bolding
- wide ASCII art
- terminal color assumptions leaking into markdown

## 6. Playbook Schema Improvements

The schema changes should improve inspectability without hiding logic in prose.

### 6.1 Rule Labels And Notes

Current `match.any`, `match.all`, and `match.none` lists are too opaque for trace output. Allow rule objects alongside strings.

Before:

```yaml
match:
  any:
    - "exec /__e/node20/bin/node: no such file or directory"
    - "no such file or directory"
  none:
    - "permission denied"
```

After:

```yaml
match:
  any:
    - id: missing_runtime_path
      pattern: "exec /__e/node20/bin/node: no such file or directory"
      label: Missing bundled runtime path
      note: Matches the GitHub Actions runtime shim path used by this failure.
      relevance: strong
    - id: generic_path_absence
      pattern: "no such file or directory"
      label: Generic path absence
      relevance: supporting
  none:
    - id: permission_denied_variant
      pattern: "permission denied"
      label: Permission-denied variant
      note: Excludes the nearby permission failure family.
```

Benefits:

- better trace readability
- explicit rule identity for snapshots and JSON trace
- better overlap diagnostics

### 6.2 Structured Remediation

Keep `fix` for backward compatibility, but add optional structured remediation fields that the renderer can prefer when present.

```yaml
remediation:
  primary:
    - summary: Install the missing runtime or tool in the CI image.
      commands:
        - "command -v <tool>"
      confidence: high
  investigate:
    - summary: Check whether the workflow or wrapper uses a hard-coded runtime path.
      preconditions:
        - The binary exists locally but not in CI.
      risks:
        - Fixing the path may expose a runner-image drift problem elsewhere.
  verify:
    - "command -v <tool>"
    - "re-run the failing workflow step"
```

Benefits:

- better operator flow
- cleaner `fix` and `workflow` rendering
- clearer copy-paste command extraction

### 6.3 Related Playbooks And Alternatives

```yaml
related_playbooks:
  - id: arch-mismatch
    relation: alternative
    note: Similar runtime startup failure, but requires architecture evidence.
  - id: runtime-mismatch
    relation: neighbor
    note: Similar failure family with version-conflict wording.
```

Benefits:

- better `explain`
- better trace and compare output
- helps authors reason about overlap intentionally

### 6.4 Provenance And Review Metadata

```yaml
metadata:
  schema_version: v0.3
  owner: runtime
  reviewed_at: 2026-04-18
  references:
    - docs/failures/github-actions/exec-node20-bin-node-no-such-file-or-directory.md
```

Benefits:

- authoring accountability
- easier preview and review tools
- better markdown provenance section

### 6.5 Disproof Checks

Current hypothesis fields already support disproof concepts. Surface that more directly in authoring.

```yaml
disproof_checks:
  - Run `uname -m` inside the failing runner to confirm this is not an architecture mismatch.
  - Search the log for `permission denied` to rule out filesystem permission failures.
```

Benefits:

- sharper differential diagnosis
- better operator guidance when confidence is moderate

## 7. Operator Workflows

### 7.1 Fast Triage

Goal:
- answer "what broke and what do I do first" in under a minute

```bash
faultline analyze ci.log
faultline fix ci.log
faultline workflow ci.log
```

Expected experience:

- `analyze` gives one-screen summary plus top evidence
- `fix` gives immediate primary action
- `workflow` gives follow-through and verification

### 7.2 Deep Investigation

Goal:
- inspect a close call or defend the diagnosis before acting

```bash
faultline analyze ci.log --top 3 --show-evidence
faultline analyze ci.log --select 2 --view summary
faultline trace ci.log --show-rejected --show-scoring
faultline explain missing-executable
```

Expected experience:

- fast candidate switching
- trace shows exact rule decisions
- rejected alternatives explain the boundary

### 7.3 Remediation

Goal:
- move from diagnosis to safe action

```bash
faultline fix ci.log --with-preconditions --with-risks
faultline workflow ci.log --checklist
faultline workflow ci.log --commands-only
```

Expected experience:

- primary fix separated from investigation paths
- commands are copy-paste friendly
- verification is explicit

### 7.4 Playbook Authoring And Debugging

Goal:
- make playbooks feel testable and traceable

```bash
faultline playbook lint playbooks/bundled
faultline playbook test playbooks/bundled fixtures/real/*.yaml
faultline playbook trace missing-executable examples/missing-executable.log
faultline playbook preview missing-executable
faultline playbook doctor playbooks/bundled
```

Expected experience:

- lint catches structural and narrative problems
- trace makes rule tuning concrete
- doctor highlights overlap, weak evidence, and poor verification guidance

### 7.5 Compare And Replay

Goal:
- explain what changed over time

```bash
faultline analyze ci.log --json > current.analysis.json
faultline replay current.analysis.json --format markdown
faultline compare previous.analysis.json current.analysis.json --why-changed
```

Expected experience:

- stable rerendering from saved artifacts
- deterministic diagnosis deltas
- new evidence isolated from unchanged noise

## 9. Risks And Tradeoffs

### Feature Creep

Risk:
- too many subcommands can dilute the default story

Mitigation:
- keep `analyze`, `workflow`, `list`, `explain`, and `fix` as the homepage
- treat `trace`, `compare`, and `playbook` as companion surfaces

### TUI Temptation

Risk:
- a full-screen UI adds complexity, terminal fragility, and maintenance cost

Mitigation:
- do not make a TUI the default interface
- keep interactivity as an optional, pager-like layer
- every interaction must map to a normal command

### Noisy Output

Risk:
- transparency features can turn into log spam

Mitigation:
- progressive disclosure by default
- `summary` remains compact
- `trace` remains explicit

### Maintainability Burden

Risk:
- multiple renderers and view modes can diverge

Mitigation:
- define one internal report model and render from it
- use snapshot tests for terminal, markdown, trace, and compare output

### Performance Regressions

Risk:
- rejected-candidate inspection and trace capture can slow analysis

Mitigation:
- keep trace data shallow by default
- compute deeper trace detail only when requested
- preserve current fast path for plain `analyze`

### Schema Drift

Risk:
- richer playbook fields may become half-adopted and inconsistent

Mitigation:
- keep old fields valid
- add lint rules and preview tooling before encouraging migration

## 10. Final Recommendation

Definitely build:

- `trace`
- better default terminal output
- better markdown output
- `--view`, `--select`, `--show-rejected`, `--show-evidence`, `--show-scoring`
- `replay`
- `playbook lint`, `playbook test`, `playbook preview`
- labeled rule items in playbooks
- structured remediation support with backward compatibility

Defer unless the earlier layers land cleanly:

- `compare` auto-history integration
- `playbook doctor`
- interactive inspect mode

Avoid entirely in v0.3:

- full-screen TUI as the main interface
- hiding more logic inside prose fields
- color-heavy or gimmicky terminal styling
- unstable markdown optimized only for one renderer
- feature growth that shifts the default narrative away from deterministic diagnosis and workflow handoff

The sharp v0.3 position should be:

**Faultline remains a boring, deterministic CLI at its core, but now it explains itself like a premium operator tool.**
